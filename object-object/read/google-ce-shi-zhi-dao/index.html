《Google软件测试之道》阅读归纳和总结

[TOC]

第1章 Google软件测试

1.1 质量 != 测试

质量不是测出来的，如果连测试都没有做，如何保证成果具有高质量？答案是将开发和测试融为一体，大家并肩同行。需要在写完一段代码后立即测试这段代码，测试不是独立隔离的环节，它本身就是开发过程的一部分。质量不等于测试，可以将开发和测试放到一起，做到相互融合，部分彼此时，质量就出来了。

那么问题来了，关键是这个阶段由谁来做这些测试工作？由开发同学负责做这些测试工作，还有谁能比写代码的人更适合找到 bug 呢，这就是我们工作中强调版本提测前要求开发先完成自测。如果某个产品质量出了问题，第一个跳出来的肯定是对应的开发，而不是遗漏这个 bug 的测试人员。

在项目管理质量管理知识领域中说到，预防(防开发过程质量问题)胜于检查(最终产品的测试)。质量是开发过程中的问题，而不是测试问题。测试的目的就是判断这种预防工作是否做到位。测试是开发过程必不可少的一部分，当开发和测试一起携手共进之时，就是质量达成之时。

1.2 角色介绍

“解铃还须系铃人” ，“you build it,you break it,you fix it” ，Bug 只有开发自己才能修复！

1.2.1 软件测试工程师（SWE）

软件开发工程师（software engineer），他们的工作是实现最终用户所使用的功能代码。他们的工作还包括设计文档、选择最优方案、代码 review 等。

1.2.2 软件测试开发工程师（SET）

软件测试开发工程师（software engineer in test）也是一个开发角色，工作重心在可测试性和通用测试基础框架上。包括参与设计评审，审查代码质量与风险，SET 的目的是为了质量服务，SWE 更注重产品功能实现。

1.2.3 测试工程师（TE）

测试工程师（test engineer）是一个和 SET 关系密切的角色，关注点在用户，站在用户的角度来思考质量方面的各种问题。TE组织整体质量实践，分析解释测试运行结果，驱动测试执行，构建端到端的自动化测试。

作为测试人，有自己选择决定优先级，在可靠性和安全性上不能妥协，除非碰到更重要的事情。

1.3 组织结构

一般看到的团队管理者，都是来自产品经理或开发经理，而不是来自测试团队。所以在产品发布时，优先考虑的是功能的完整性和易用性，很少考虑质量问题，这导致测试总是给开发让路。

测试一般是独立存在的部门，测试人员被借调到产品团队中，去做提高产品质量相关的事情。一个团队不能随意降低测试人员的技术要求，从而招聘更多的测试人员，然后让他们做一些简单和琐碎的事情。功能相关的脏活累活本应该是开发的工作，不能简单地掉给测试，但实际上我们都搞错了。

测试人员可以在不同项目之间借调，或者隔一两年进行自愿换岗，可以让 SET 和 SE 保持新鲜感并且保持忙碌。另外还能保证一个好的测试想法可以在公司内部蔓延。

1.4 爬、走、跑

介绍了通过不同开发版本的不断迭代，不断优化最终对外发布。

1.5 测试类型

Google 并没有使用单元测试、集成测试、系统测试等这些命名方式，而是通过小型、中型、大型测试来称呼，着重强调测试的范畴规模而非形式。测试类型的描述属于是什么并不是重点，怎么命名都可以，重要的是组织内部使用的术语要统一，最好是按照字面意思就可以理解是最好的。

第2章 软件测试开发工程师

融合开发角色和质量意识于一身的角色。

2.1 SET 的工作

2.1.1 测试和开发的流程

SET 会参与到许多测试目标的构建中，并指出哪些地方需要小型测试。在多个构建目标集成在一起后，开始做集成测试。当构建目标日益增大时，SET 也会参与到中大型测试的编写中。

SET始终是活动的核心参与者，当开发不知道哪些地方需要做单元测试的时候可以明确指出。SET 同时编写许多mock 和 fake 工具，甚至编写中大型集成测试。

2.1.2 SET 究竟是谁

首先是工程师角色，使得测试尽早介入到开发流程中。积极参与各种测试，提高测试效率。SET 和 功能开发人员处于相同地位，大家都在相互交流学习，形成最有效率的产品团队。

2.1.3 项目的早期阶段

项目初期，不会投入太多的测试资源，在未来可能失败的项目中投入测试资源构建测试方面的基础设施，这是一种资源的浪费。如果项目被取消，之前的所有工作都会变得毫无意义。即使SET在早期参与到项目中来，也不会是做测试工作，而是做开发。

尤其是在项目还处在概念阶段时，测试不会介入，如果代码需要重构，那么投入的大量测试和可测试性方面的努力，其实没有太大的实际价值。项目立项后，就需要发挥我们测试人如何执行测试的影响力。

2.1.4 团队结构

SWE 仅在自己的模块领域提供最优方案，但如果从产品的角度来看，视野会显得狭窄。一个好的 SET 正好可以弥补这一点，不仅要具备整体产品视野，而且在产品整个生命周期里对产品及功能特性做充分理解。

产品团队初期的技术负责人，一般是由工程师担任，负责设定技术方向、开展合作、充当与其他团队沟通的对接人。项目技术负责人和发起人要做的第一件事就是设计文档，设计文档不断完善，不同专业类型的工程师就会投入到项目中。

2.1.5 设计文档

在设计阶段，SET 在推进项目的同时也可以简化相关项目成员的工作。SET 需要对设计文档进行审核，在审核过程中，会针对质量和可靠性方面增加一些必要的内容。这么做有一下几个原因：

  • SET 需要熟悉了解所有负责的系统设计。
  • SET 早期提出的建议会反馈在文档和代码中，这样也会增加 SET 的整体影响力。
  • 作为第一个审阅所有设计文档的人，包括所有迭代过程，SET 对项目的整体了解程度超过技术负责人。
  • 对于 SET 来说，可以更早的和 SWE 建立良好的工作关系。

审阅设计文档是应该保持强烈的目的性，主要体现在以下几点：

  • 完整性：找出文档中残缺不全的点或存在一些知识盲区，鼓励文档设计者添加更多的细节、增加一些关联文档链接等。
  • 正确性：检查文档格式、错别字、语法、标点符号等错误，以及业务描述的正确性。
  • 一致性：确保配图和文字描述的一致性。
  • 设计： 文档中的一些设计要经过深思熟虑。考虑可用资源、用什么框架、设计是否过于复杂、目标是否能顺利达成等等。
  • 接口与协议：文档中对所用的协议是否有清晰的定义？接口协议是否符合调用方的要求？使用标准是否统一？
  • 测试：系统和文档中描述的整套系统的可测试性怎样？系统设计是否考虑到易测性？

2.1.7 自动化计划

在端到端自动化测试上多读投入，尝尝会把你与产品的特定功能绑定设计在一起。在谷歌，SET 首先把容易出错的接口做隔离，并针对它们创建 mock,确保测试覆盖率。然后构建一个轻量级的自动化框架，控制 mock 系统的创建和执行。

SET 除了这个计划中涵盖自动化之外，还要包括如何展示产品质量方面的信息给相关人员。如报表、仪表盘等展示测试结果和测试进度。通过将整个过程简单化和透明化，有助代码质量的提高。

2.1.8 可测试性

SET 编写测试框架，为 SEW 编写测试代码方面提供帮助，同时提供一些程序接口和代码风格方面的建议给开发人员，让他们可以更好的做单元测试。

2.1.10 执行测试

测试自动化不仅仅是自动化测试代码的编写，也必须要考虑如何编译测试代码、执行、分析、存储和报告所有测试运行结果，如何让自动化测试更具有价值。

2.1.11 测试大小的定义

谷歌中分为三种测试方式：

  • 小型测试：通常就是单元测试，为了验证一个代码单元的功能，一般只涉及单一函数。
  • 中型测试：通常我们叫集成测试，验证两个或多个模块应用之间的交互。运行测试时间比小型测试更久，所以需要控制执行频率，一般由 SET 组织执行中型测试。鼓励使用 mock 技术来解决外部服务的依赖问题。
  • 大型测试：谷歌之外通常叫系统测试或者端到端测试，验证系统整体是如何工作的。
2.1.12 测试规模在共享测试平台中的使用

对一个大型的组织来说，想用一个通用的测试平台，那么这个平台必须能做各种各样的测试任务。通过测试用例的耗时，定义测试类型。调度器利用每个测试任务需要运行的时间，优化任务队列，达到合理利用的目的。

2.1.13 测试规模的益处

介绍了小、中及大型测试的优缺点，如何检验一个项目中小、中、大型测试之间的比例是否健康。

2.1.14 测试运行要求

如果执行测试系统是一个公用环境，要求测试本身满足以下条件：

  • 测试和测试之间相互独立
  • 测试不做任何数据持久化方面的工作。保证测试环境在测试结束后和开始前环境一致。
  • 等等

维护好测试系统，创建一系列文档，形成“测试百科全书”。

3 测试工程师

TE 重点在于评估对用户的影响以及软件产品整体目标上的风险。

3.1 一种面向用户的测试角色

TE 综合了开发者仰慕的技术能力和以用户为中心检查软件质量而对开发者产生一定制约的能力。TE 的工作难描述，因为指责范围广不好确定。都希望 TE 完成所有成果研发过程中的质量，所以往往 TE 都是从事一些基础技术层的。

3.2 测试工程师的工作

在开发的前期阶段，TE 通常没有太多的工作可做，因为很多功能可能会存在变化，甚至可能会改需求。TE 介入产品的时候，不需要从零开始。SWE 和 SET 已经在测试技术和质量方面做了大量的工作，可以作为 TE 的起点。TE 介入产品时需要注意一下一些问题。

  • 当前软件的薄弱点在哪里？
  • 有没有安全、隐私、性能、可靠性、可用性、兼容性、全球化和其他方面的问题？
  • 主要用户场景功能是否正常？对于全世界不同国家的用户都是这样吗？
  • 这款产品能与其他产品（软硬件）互操作吗？
  • 当发生问题时，是否容易排查问题所在？如打印适量的日志。

当然以上所列还不全，所有这些加起来可以构成发布待评估软件的风险概要。TE 保证以上这些问题被解决，可以找他人评估还有什么工作需要进行。TE 的根本使命就是保护用户和业务的利益。

成功的 TE 游走于一些微妙且敏感的地方，通过测试找出bug。TE 通常是团队中最出名的人，因为需要与各种角色沟通。

下面是关于 TE 职责的一般性描述。

  • 测试计划和风险分析。
  • 评审需求、概设、代码和测试。
  • 探索式测试。（边学习、边设计、边测试、边思考）
  • 用户场景。
  • 编写测试用例。
  • 执行测试用例。
  • 使用统计。
  • 用户反馈。
3.2.1 测试计划

测试计划最早出现，也是最先被遗忘的测试产出。下面是测试计划具体的一些特性。

  • 及时更新。
  • 描述了软件的目标和卖点。
  • 描述了软件的结构、各种组件和功能特性的名称。
  • 描述了软件的功能和操作简介。
    从测试的角度看，我们担心的是测试计划的投入和价值产出是否匹配。
  • 不必花过多的时间去编写，必须随时可以被修改。
  • 应该描述必测点。
  • 应能在测试中提取有用的信息，从而帮助确定进展以及覆盖率上的不足。

知道原则如下。

  • 简明。不需要过多描述词。

  • 只写相关人员关心的内容，重要的内容和可执行的内容。

  • 渐进式描述。

  • 指导计划者的思路。

  • 最终结果应该是测试用例。

做出一个不直接知道测试的计划就是在浪费时间。如果测试计划没有吧测试用例应该怎么执行描述得足够详细，它就没有达到帮助测试的本意。

3.2.2 风险

  1. 风险分析

    软件测试中，我们可以按照常识性的过程来理解风险，下面是一些可参考的因素。

    • 哪些事件需要担心。
    • 这些事发生的概率。
    • 一旦发生，有哪些影响。
    • 产品具备什么缓解措施，这些措施失败的概率，处理这些失败的成本有哪些。
    • 恢复过程有什么困难。
    • 事件是一次性问题，还是会再次发生。

    风险分析的目标不是给出一个精确值，而是识别一个能力与另一个相比风险是大是小。

    可以针对风险发生的频率，设置预值。如，罕见、少见、偶尔、常见、较大、最大等。

    风险评估可以邀请相关人员给出一些建议。

    • 开发人员。
    • 项目经理和产品经理。
    • 销售人员。
    • 总监和 VP 等。

  2. 风险缓解

    首先要知道，风险不大可能彻底消除。有很多措施可以缓解风险。

    • 我们可以围绕风险大的功能点写用户故事。
    • 编写回归测试用例。
    • 编写和运行引发故障的测试用例。
    • 添加监控。

    原则上，如果不能全测，就先测风险大的功能。
